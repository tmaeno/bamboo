# Docs platform pilot ÔÇö Astro Starlight

This repository was the **pilot** for choosing a replacement docs platform after Material for
MkDocs entered maintenance mode (Nov 2025) and MkDocs core became effectively unmaintained.
The goal was to evaluate feasibility, performance, presentation, and migration friction so the
findings can drive migrating the rest of the fleet.

The docs now live in [`website/`](website/) and build with Astro Starlight.

## Verdict: GO

Starlight is a good fit for the fleet. The migration of this repo was mechanical and fully
verified (clean build, all links valid, diagrams render, OpenAPI reference hosted). Nothing
encountered would block rolling it out to other repos. The main standing costs are a Node
toolchain on otherwise-Python repos (already acceptable ÔÇö Redocly is Node too) and a few
one-time conversions per repo (see *Per-repo effort*).

## What was migrated

- 20 Markdown pages (13 root + 7 under `database-plugins/`) Ôćĺ `website/src/content/docs/`,
  reorganized into nav folders (`guides/`, `integrations/`, `architecture/`, `database-plugins/`).
- 75 internal cross-document links rewritten to root-absolute URLs.
- 10 Mermaid diagrams across 7 pages.
- The home page rebuilt as a Starlight **splash hero** with `CardGrid`/`LinkCard` and a
  role-based `Tabs` block; a `:::tip` aside added to Quick Start (to exercise rich components).
- OpenAPI reference generated by **Redocly CLI** Ôćĺ hosted at `/bamboo/api/`, linked from the sidebar.
- Deploy workflow (`.github/workflows/docs.yml`) rewritten for Node/Astro Ôćĺ GitHub Pages.
- MkDocs removed: `mkdocs.yml`, the `[docs]` extras in `pyproject.toml`, old `docs/` tree.

## Performance (this repo, 21 pages, local ÔÇö Node 24, M-series Mac)

| Metric | Value |
|---|---|
| Cold production build (Astro only) | ~6.1 s |
| Cold build incl. Redocly + Pagefind (`npm run build`, wall clock) | ~8.6 s |
| Dev server startup (`npm run dev`, ready) | ~1.9 s |
| Total build output (`dist/`) | 5.9 MB |
| Ôć│ `_astro/` JS+CSS | 3.4 MB (**dominated by the Mermaid client bundle**) |
| Ôć│ `pagefind/` search index | 852 KB |
| Ôć│ `api/` (Redocly HTML) | 52 KB |

**Performance note:** the client payload is dominated by Mermaid (~3 MB). For diagram-heavy or
perf-sensitive repos, switching to build-time **`@beoe/rehype-mermaid`** (emits static SVG,
zero client JS) would cut that substantially ÔÇö at the cost of a headless-browser (Playwright)
dependency in CI. For this pilot, client-side rendering was kept for build simplicity.

## Fleet rollout guidance

**Reusable across repos:**
- The **link-rewrite codemod** (a ~80-line Node script keyed by a `basename Ôćĺ new URL` map) ÔÇö
  handles the title-from-H1 + link rewrite mechanically.
- The `astro.config.mjs` pattern (base path, sidebar shape, `astro-mermaid` + `starlight-links-validator`).
- The `.github/workflows/docs.yml` NodeÔćĺPages workflow.
- The Redocly-into-`public/api/` hosting pattern (just repoint `docs:api` at each repo's real spec).

**Per-repo effort:**
- *Plain-Markdown repos like this one:* a few hours ÔÇö mostly building the basenameÔćĺURL map and the sidebar.
- *Repos using MkDocs/pymdownx features* (admonitions, content tabs, snippet includes, `attr_list`):
  add conversions to the codemod ÔÇö `!!! note` Ôćĺ `:::note`, `=== "Tab"` Ôćĺ `<Tabs>`, `--8<--` Ôćĺ `<!-- @include -->`/imports. This repo had none, so that path is untested here and should be budgeted for.

**Caveats to flag fleet-wide:**
- `title:` is mandatory per page; internal links must include the base prefix.
- Mermaid client-bundle weight (consider build-time SVG for heavy users).
- No built-in doc **versioning** (Starlight needs a community plugin); revisit if a repo needs versioned docs.
